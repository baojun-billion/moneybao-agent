## 阶段反思（19:27）

**时间**: 2026-04-21 19:27
**状况**: 访客端语音识别由于 
ot-allowed 报错无法使用。

**原因分析**:
1. 浏览器的 SpeechRecognition API (Web Speech API) 要求网站必须在 HTTPS 下运行，或者用户必须明确授予麦克风权限。如果在普通的 HTTP 环境（非 localhost，例如通过 IP 地址访问局域网），浏览器会自动阻断麦克风权限并返回 
ot-allowed 错误。
2. 虽然之前写了组件测试并全部通过，但组件测试主要是针对 UI 渲染和函数调用。这种底层浏览器的安全策略限制无法通过纯 UI 测试暴露出来，需要到真实的端到端环境（或者了解其限制）才能发现。
3. 作为 C端功能，不能因为麦克风权限问题卡死整个交互。如果没有语音权限，必须提供一个文本输入框（Fallback）作为补充手段。

**解决方案**:
1. 在 Reception.tsx 中添加文本输入框功能，允许用户在语音识别不可用时打字交流。
2. 捕获权限错误并给出人性化的提示。

## 阶段反思（20:05）

**时间**: 2026-04-21 20:05
**状况**: 用户强调麦克风功能是非常重要的应用要求，且用户表示可以在本地访问（即在运行代码的电脑上直接使用 localhost）。

**原因分析**:
1. 之前我误以为用户是在手机上通过局域网 IP 访问才导致的 
ot-allowed，但用户说他可以本地访问（意味着在 Windows 电脑上打开浏览器）。
2. 如果是在电脑的 localhost 上访问依然出现 
ot-allowed，那说明**根本原因不在于 HTTPS 限制**（因为 localhost 是被浏览器豁免的安全上下文），而在于电脑或浏览器的系统级麦克风权限问题。

**导致 
ot-allowed 的真正可能原因（在 localhost 下）**:
1. **浏览器自身拒绝**: 之前弹出过麦克风授权提示框，被用户不小心点击了"拒绝（Block）"。
2. **Windows 系统级隐私设置拦截**: Windows 10/11 的"隐私与安全性" -> "麦克风"中，关闭了"允许应用访问你的麦克风"或"允许桌面应用访问你的麦克风"。
3. **硬件问题**: 电脑根本没有连接麦克风，或者麦克风驱动异常，导致浏览器调用 API 失败并返回 
ot-allowed。

**推进策略**:
既然语音交互是核心功能，且代码层面已经使用了原生的 Web Speech API（这本身在支持的浏览器如 Chrome 中是非常可靠的），当前的重点不是改代码，而是**排查和解决宿主机（包哥的 Windows 电脑）的麦克风授权与配置问题**。我需要给出清晰的一步步排查指南。

## 阶段反思（20:10）

**时间**: 2026-04-21 20:10
**状况**: 语音和文字均返回 "抱歉，服务器出现了点小问题，请稍后再试。"

**原因分析**:
1. 这是通过 ConsultationService 中的 catch 块返回的标准降级回复。说明后端代码抛出了错误。
2. 刚刚测试了后端的 text 接口，发现具体错误是 Failed to generate reply: Request failed with status code 401。
3. HTTP 401 意味着 **未授权 (Unauthorized)**。这个请求是后端向 GLM-4 的 API （https://open.bigmodel.cn/api/paas/v4/chat/completions）发出的。
4. 这说明，.env 文件中的 GLM4_API_KEY 是无效的（目前使用的是默认的 your_glm4_api_key_here 或者没有填写真实的 Key）。GLM-4 官方拒绝了我们的请求。

**解决方案**:
需要让宿主（包哥）在后端的 .env 文件中填写真实有效的智谱 GLM-4 API Key。
